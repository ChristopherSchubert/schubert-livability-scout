# Six blocks

A short, curated list of specific blocks / wharves / streets to walk in
each city — the "where to stand to see what this place actually feels
like" list. Five to seven entries per city, each a single sentence-or-
less location string.

## How it works today

- **Data**: `cities.blocks` jsonb — an array of strings like
  `"Thames Street at Howard Wharf"`, `"Spring Street between Touro & Mill"`.
  Populated at insert time (UI *+ Add candidate* or
  [scripts/onboard.mjs](../scripts/onboard.mjs)) and lives on the row.
- **Per-block images**: `cityItem.blockImages` mirrors the array shape
  with one image query per block (auto-generated by
  `blockImageQuery(cityName, block)` in
  [lib/planner-data.js](../lib/planner-data.js)).
- **Render (live)**: **not wired**. Grepping the components for
  `cityItem.blocks` returns nothing — the data is on the row, the
  factory writes it, but no current React component reads it.
- **Render (mockup)**: chapter 6 ("Six blocks") of
  [public/city-detail-redesign.html](../public/city-detail-redesign.html)
  shows the intended UI — one card per block, with the block image and
  the location string.

## Status

- **Data layer**: populated for most cities (entries vary in count and
  quality — some are 5-block lists, some are partial).
- **UI layer**: zero. The mockup is the only place blocks render.

## TODOs / future direction

- **Wire chapter 6 into the live route.** One card per block + matching
  image. This is the most concretely "ready to ship" of the mockup
  chapters — the data is already there.
- **Block image search.** Today the query is auto-generated. A search /
  paste-URL flow per block (mirroring the hero image pattern) would let
  the owner curate. Without it, block images are best-effort and often
  off-target.
- **Block validation.** No source check that the block actually exists —
  a typo silently lives on the row. Consider verifying each block string
  via Nominatim at insert time and flagging unresolvable ones.
- **Re-ordering / pruning.** Today block order is insertion order; the
  owner can't reorder without editing the row. A small drag-to-reorder
  UI alongside the curate flow would help.
- **Tie blocks to the in-trip Journal mode** (TODO #6) — when the owner
  is in the city, each block gets a tap-to-rate entry.
